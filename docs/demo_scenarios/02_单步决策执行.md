# 场景 2：单步决策执行

## 场景目标

演示调度器的单步决策过程。通过停止自动调度后连续执行 `step`，观察每一步选择哪个进程、运行了多少时间片、是否发生队列迁移，以及总览页面如何反映这些变化。

## 核心操作

1. 登录已有演示用户 `demo`。
2. 停止自动调度，确保演示由手动单步驱动。
3. 创建三个不同优先级的进程。
4. 查看就绪队列。
5. 连续执行三次单步调度。
6. 查看系统总览。

## 菜单操作路径

菜单名称以当前 `MenuConsole` 为准：

1. 用户管理 → 登录用户
2. 调度管理 → 停止自动调度
3. 进程管理 → 创建进程
4. 进程管理 → 查看就绪队列
5. 调度管理 → 单步调度
6. 调度管理 → 单步调度
7. 调度管理 → 单步调度
8. 系统总览 → 查看系统全局总览

## 原始命令脚本

```text
login demo 123456
stop_sched
create_pcb step_a 64 0 8
create_pcb step_b 64 5 8
create_pcb step_c 64 10 8
readyq
step
step
step
overview
```

## 预期输出

- `stop_sched` 返回停止调度器的结果；如果调度器原本未运行，也应保持可继续操作。
- `readyq` 展示三个进程所在队列。
- 每次 `step` 输出本次选择的 PID、运行 tick 和队列变化。
- `overview` 的进程表展示 CPU 进度，MLFQ 区域展示当前就绪队列快照。

## 验收标准

- 三次 `step` 均返回真实调度日志。
- 至少一个进程的 CPU 时间发生变化。
- `overview` 中的进程状态、CPU 进度和队列展示与单步结果相符。
- 整个流程不依赖自动调度线程持续运行。

## 截图建议

- 截取 `readyq` 初始输出。
- 截取连续三次 `step` 的调度日志。
- 截取 `overview` 中 Process Table 和 MLFQ 两个区域。

## 常见问题排查

- 如果 `login demo 123456` 失败，先运行场景 1 的注册步骤，或手动执行 `register demo 123456`。
- 如果就绪队列为空，检查进程是否已被之前的演示完成或删除。
- 如果自动调度干扰单步结果，先再次执行 `stop_sched`。
