# 场景 3：状态持久化

## 场景目标

演示系统状态保存、退出、重启后的恢复效果。重点验证用户、进程、调度后的运行状态和虚拟文件系统内容能够通过快照保存并重新加载。

## 核心操作

1. 注册并登录 `persist` 用户。
2. 创建父子进程。
3. 写入虚拟文件。
4. 执行两次单步调度，让进程状态产生变化。
5. 查看总览并保存系统状态。
6. 退出程序，重启后登录同一用户。
7. 查看进程、总览和文件内容。

## 菜单操作路径

菜单名称以当前 `MenuConsole` 为准：

1. 用户管理 → 注册用户
2. 用户管理 → 登录用户
3. 进程管理 → 创建进程
4. 虚拟文件系统 → 写入文件
5. 调度管理 → 单步调度
6. 系统总览 → 查看系统全局总览
7. 持久化管理 → 保存系统状态
8. 退出程序后重新启动
9. 用户管理 → 登录用户
10. 进程管理 → 查看进程列表
11. 系统总览 → 查看系统全局总览
12. 虚拟文件系统 → 读取文件

## 原始命令脚本

首次运行：

```text
register persist 123456
login persist 123456
create_pcb persist_init 64 0 20
create_pcb persist_worker 128 5 15 1
write_file note.txt persistent data
step
step
overview
save
exit
```

重启后运行：

```text
login persist 123456
list_pcb
overview
read_file note.txt
```

## 预期输出

- 首次运行时 `save` 输出保存成功，并显示保存的进程、用户或文件统计信息。
- 重启后 `login persist 123456` 成功。
- `list_pcb` 能看到 `persist_init` 和 `persist_worker`。
- `overview` 能展示恢复后的进程树、内存图和 MLFQ。
- `read_file note.txt` 输出 `persistent data`。

## 验收标准

- 保存前后的进程名称一致。
- 文件 `note.txt` 的内容在重启后仍可读取。
- 快照加载不要求保留登录态；重启后需要重新登录是正常行为。
- 演示数据来自实际保存和加载，不硬编码输出。

## 截图建议

- 截取保存前的 `overview`。
- 截取 `save` 成功输出。
- 截取重启后的 `list_pcb` 和 `read_file note.txt`。

## 常见问题排查

- 如果重启后提示未登录，先执行 `login persist 123456`。
- 如果数据没有恢复，确认上一次退出前已经执行 `save`。
- 如果快照文件来自旧演示，可先使用新用户名避免和历史数据混淆。
