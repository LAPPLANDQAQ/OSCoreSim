## Module Findings

### Clean Boot

True clean boot was verified by running the executable from a temporary working directory that had no `data/os_state.bin`.

Observed clean result:

- `Auto Load: no state file`
- `create_pcb clean_init ...` produced `PID=1`
- Registration and login succeeded

Running the same script from the repository root loads the existing snapshot:

- `Auto Load: success`
- `create_pcb clean_init ...` produced `PID=9`

Conclusion: PID does start from 1 only when no snapshot is present. The current repository runtime state is not clean.

### User and Session

Observed behavior matches current implementation:

- `register p1user abc` succeeds
- `login p1user abc` succeeds
- second login while already logged in fails
- `logout` clears the session
- three bad login attempts lock `p1lock`
- correct password after lock is rejected

No user/session regression was reproduced aside from expected locked-account behavior.

### Process

Current snapshot imports `nextPid = 9`, so the process test uses PIDs 9, 10, and 11 for the positive lifecycle path.

Positive commands succeeded:

- `create_pcb`
- `list_pcb`
- `show_pcb 9`
- `show_pcb 10`
- `ptree`
- `readyq`
- `block_pcb 10`
- `wakeup_pcb 10`
- `suspend 11`
- `resume 11`
- `renice 11 2`
- `kill_pcb 9`

Failure reproduced:

- `show_pcb 1` fails because the current snapshot does not create PID 1 for the diagnostic user.

### Memory

The current snapshot allocation algorithm is `WORST_FIT`, with existing Bob-owned memory blocks. This changes manual allocation addresses.

Positive commands succeeded:

- `alloc p1mem_a 40` produced start `536KB`
- `alloc p1mem_b 20` produced start `576KB`
- `free_mem 576` succeeded
- `set_alloc_algo BF` and `set_alloc_algo WF` succeeded
- `compact` succeeded and moved existing blocks
- `pgfault` generic succeeded
- `pgfault 9` succeeded after creating PID 9
- `swap_out 9` succeeded

Failures reproduced:

- `free_mem 0` failed because address 0 is a free block in the loaded snapshot
- `pgfault 1` and `swap_out 1` failed because PID 1 is not owned by the diagnostic user

### Scheduler

Canonical scheduler commands work:

- `step`
- `start_sched`
- `stop_sched`
- `restart_sched`

Alias probes fail:

- `start`
- `stop`
- `restart`

Conclusion: scheduler recognition is correct for documented canonical names, but any callers/tests expecting shorthand names will fail.

### Persistence

Positive behavior:

- `save` succeeds
- `load` succeeds
- process state persists
- after re-login, `create_pcb p1persist_after_load ...` produces `PID=10`, confirming `nextPid` advances from saved `PID=9`

Failure/surprise behavior:

- `load` clears the current session
- `whoami` after `load` returns `not logged in`
- `list_pcb` and `create_pcb` after `load` fail until login

Conclusion: persistence restores system state but intentionally does not restore interactive login state.

### VFS

- `read_file p1_note.txt`

Expected negative probe:

No VFS command-name regression was reproduced for canonical `_file` commands.

## Recommended Next Phase

Phase 2 should choose which behavior is intended before code changes:

1. Decide whether tests should run against a clean temporary working directory or the repository snapshot.
2. Decide whether `start`/`stop`/`restart` should become supported aliases or remain invalid.
3. Decide whether `load` should preserve the current login session or continue clearing it.
4. Replace hard-coded PID/address expectations in regression tests with either clean-state setup or output-derived expectations.
5. Only after those policy decisions, fix command compatibility or test assumptions.
