# 核心流程说明

## 1. 程序启动流程

```text
main()
  → configureConsoleEncoding()          UTF-8 控制台编码
  → InstanceGuard::initialize()         Named Mutex 选主
     → CreateMutexA("Local\OS_SIM_KERNEL_MASTER_MUTEX_2026")
     → 互斥量已存在 → CLIENT，否则 → MASTER
  → ConsoleApp::run(role)
```

**MASTER 分支：**

```text
ConsoleApp::masterLoop()
  → Kernel::start()
     → requestQueue_.reset()
     → 检查 data/os_state.bin 是否存在
        → 存在 → 加载快照 → 显示启动信息
        → 不存在 → 启动干净系统
     → 启动 worker 线程  (Kernel::workerLoop)
     → 启动调度线程     (Kernel::schedulerLoop)
  → NamedPipeServer::start(handler)
     → 启动管道服务器线程
  → 打印 MASTER Banner
  → 交互终端：显示中文数字菜单
  → 本地命令输入循环
     → 读取输入 → submitCommand() → 打印响应
  → PipeServer::stop() + Kernel::stop()
```

**CLIENT 分支：**

```text
ConsoleApp::clientLoop()
  → 不创建 Kernel，不启动调度/持久化
  → 打印 CLIENT Banner
  → 交互终端：显示中文数字菜单
  → 命令输入循环
     → NamedPipeClient::sendCommand() → 管道 → Master
     → 接收响应 → 打印
  → exit/quit 仅关闭 Client 自身
```

## 2. 命令执行流程

```text
ConsoleApp 前台线程
  → std::getline(input, line)
  → Kernel::submitCommand(line)
     → 包装为 CommandRequest（含 std::promise）
     → requestQueue_.push(request)
  → future.get() 阻塞等待

Kernel worker 后台线程
  → requestQueue_.pop(request)
  → executeRequest(request)
     → 解析命令名
     → 路由判断：
        isSchedulerCommand    → handleSchedulerCommand()
        isPersistenceCommand  → handlePersistenceCommand()
        isVisualizationCommand → handleOverview()
        isVfsCommand          → handleVfsCommand()
        否则                   → dispatcher_.dispatch()
     → 返回 CommandResponse
  → request.promise->set_value(response)
```

## 3. create_pcb 创建进程流程

```text
用户: create_pcb init 64 0 20 [ppid]
  → CommandDispatcher::dispatch()
  → requireLogin() 检查登录
  → 解析参数：name, memKB, priority, totalTime, ppid
  → MemoryManager::allocateForProcess(owner, expectedPid, name, 64KB)
     → FF/BF/WF 算法搜索空闲块
     → 分配物理内存块
  → ProcessManager::createProcessWithMemory(owner, name, 64KB, memStart, 0, 20, ppid)
     → 新建 PCB：分配 PID, 设置 READY 状态, 计算队列等级
     → 插入 pcbTable_
     → 维护父进程 children 列表
     → 入队 readyQueues_[queueLevel]
  → 如果 PCB 创建失败 → 回滚已分配的内存
```

## 4. step 单步调度流程

```text
Scheduler::step(owner, processManager, memoryManager)
  → [调度前] 打印 ready queues 快照
  → ProcessManager::cleanupInvalidReadyQueueEntries()
     → 清理 not READY / swappedOut 的无效条目
  → ProcessManager::pickNextReadyProcess(owner)
     → 从 Q0 → Q1 → Q2 扫描
     → 跳过非当前用户的进程
     → 返回第一个 READY 进程的 PID
  → 若无就绪进程 → 输出 "CPU 空闲"
  → ProcessManager::markRunning(pid)
     → 从 ready queue 移除
     → 状态 → RUNNING
  → ProcessManager::tickProcess(pid, quantum)
     → 逐 tick 推进 executedTime, remainingTime, timeSliceLeft
  → [结果] 判断：
     → remainingTime == 0 → 进程完成
        → killProcess() → 释放 PCB + 内存
     → 时间片耗尽 → demoteProcess() + markReady() → 降级入队
     → 时间片未耗尽 → markReady() → 原级入队
  → [调度后] 打印新的 ready queues 快照
```

## 5. start_sched 自动调度流程

```text
Kernel::schedulerLoop()
  → while (!shuttingDown_)
     → 检查 schedulerRunning_ (atomic<bool>)
     → 未运行 → sleep 100ms → continue
     → 已运行 → lock(stateMutex_)
        → scheduler_.step(owner, processManager, memoryManager)
     → 打印调度日志到控制台
     → sleep 500ms
```

## 6. alloc 内存分配流程

```text
MemoryManager::allocateManual(owner, sizeKB)
  → allocateLocked(owner, 0, "manual", sizeKB, KERNEL, ...)
     → findFreeBlockLocked(sizeKB)
        → FF: 返回第一个足够大的 FREE 块
        → BF: 返回最小的足够大的 FREE 块
        → WF: 返回最大的 FREE 块
     → 分割空闲块（或完全占用）
     → sortBlocksLocked()
  → 返回起始地址和成功信息
```

## 7. compact 内存紧缩流程

```text
MemoryManager::compact()
  → 收集所有非 FREE 块到 allocated 列表
  → 按原始顺序排列，从地址 0 开始重新分配起始地址
  → cursor 递增，记录 pidNewStart 映射
  → 在末尾添加一个 FREE 块填充剩余空间
  → Kernel 遍历 pidNewStart，调用 ProcessManager::updateProcessMemoryStart()
  → 返回移动信息和成功状态
```

## 8. save/load 二进制持久化流程

```text
save:
  → Kernel::handlePersistenceCommand("save")
  → 组装 KernelSnapshot:
     users, nextPid, pcbs, readyQueues, memoryBlocks,
     totalMemoryKB, allocAlgorithm, schedulerRunning, schedulerOwner,
     nextFileId, virtualFiles
  → SnapshotStore::save()
     → 写入临时文件 data/os_state.tmp
     → 二进制格式：
        [Magic: OSSM2026][Version: 2][HeaderSize: 20][Flags: 0]
        [Users][PCBs][ReadyQueues][Memory][Scheduler][VFS]
     → flush + rename(data/os_state.tmp → data/os_state.bin)

load:
  → SnapshotStore::load()
     → 打开 data/os_state.bin，读取文件头
     → v1 兼容：VFS 字段保持空值
     → v2：读取完整 VFS 数据
  → Kernel::importSnapshotLocked()
     → 校验：用户/PID/父子关系/内存连续性
     → 导入各子系统
     → 重建 PCB 父子关系
     → 校验 ready queues
  → 清除登录会话，停止调度器
```

## 9. Named Pipe 多实例共享流程

```text
选主：
  → InstanceGuard 创建命名互斥量
  → 首个实例 → MASTER，后续 → CLIENT

Pipe Server (MASTER 线程):
  → while running:
     → 预创建管道实例 CreateNamedPipeA("\\.\pipe\OS_SIM_PIPE_2026")
     → ConnectNamedPipe 阻塞等待客户端
     → 读取 uint32_t 长度 + raw bytes 命令
     → Kernel::submitCommand() 执行
     → 写入 uint32_t 长度 + raw bytes 响应
     → 预创建下一个管道实例（避免 WaitNamedPipeA 超时）
     → DisconnectNamedPipe + CloseHandle

Pipe Client:
  → while running:
     → WaitNamedPipeA(2000ms)
     → CreateFileA 连接管道
     → SetNamedPipeHandleState(PIPE_READMODE_MESSAGE)
     → 发送命令（长度前缀）
     → 读取响应（长度前缀）
     → CloseHandle
```

## 10. VFS 文件读写流程

```text
touch_file note.txt:
  → VirtualFileSystem::createFile(owner, "note.txt")
     → 校验文件名（1-64 字符, 字母/数字/_.-）
     → 检查同用户下是否重名
     → 创建 VirtualFile（fileId++, createdAt/modifiedAt = now）
     → 加入 files_ 列表

write_file note.txt hello world:
  → VirtualFileSystem::writeFile(owner, "note.txt", "hello world")
     → 查找已有文件 → 更新 content 和 modifiedAt
     → 不存在 → 自动创建并写入

read_file note.txt:
  → VirtualFileSystem::readFile(owner, "note.txt")
     → 查找文件 → 返回格式化文本（名称/所有者/大小/内容）

save/load:
  → VFS 数据作为 KernelSnapshot 的一部分
  → 写入：fileId, owner, name, content, createdAt, modifiedAt
  → 读取：按字段顺序反序列化
```

## 11. overview 渲染流程

```text
OverviewRenderer::render(...)
  → [1] 系统摘要
     → 用户、调度器、内存统计、进程数
  → [2] 进程树
     → getProcessCopiesForUser() 获取 PCB 副本
     → 构建 pid→PCB map
     → ppid==0 或无父进程者为根
     → 递归 appendTreeNode()（ASCII 树形）
     → visited 集合防循环
  → [3] 内存图
     → ASCII 条状图 + 明细表格
     → 其他用户块 → "OTHER_USER"
  → [4] MLFQ 队列
     → Q0/Q1/Q2 进程列表
     → 异常检测：不存在/状态异常/队列不一致/swappedOut → Warnings
  → [5] Notes
```
